# CLI

`trex` is the command-line tool for managing a Trex deployment. It is a fork of
the Supabase CLI and exposes the same command surface (`trex login`, `trex link`,
`trex functions deploy`, `trex secrets`, `trex gen types`, `trex config push`)
pointed at the [Supabase-compatible management API](apis/functions#management-api-supabase-cli-compatible)
that Trex implements.

The fork's source lives under `plugins/cli/` of the trexsql repository. Releases
are published as the npm package `trex`, plus deb/rpm/apk/archlinux packages
generated by GoReleaser.

## Install

### npm

```bash
npm install -g trex
trex --version
```

The postinstall script downloads the platform-specific binary from the matching
GitHub release.

### Build from Source

```bash
cd plugins/cli
go build -o trex
./trex --version
```

### Upstream Supabase CLI

Because the management API surface is Supabase-compatible, the upstream
`supabase` CLI (≥ v1.200) also works against a Trex server. The two binaries
share the same on-disk config layout (`supabase/config.toml`) and `SUPABASE_*`
environment variables, so you can switch between them without re-running `init`.

## Login

The CLI authenticates with a personal access token (`sbp_…`). Trex issues these
through an ECDH-sealed device-code flow:

```bash
trex login --use-api http://localhost:8001
```

The command opens your browser to the Trex login page, where an admin user
approves the request. After approval the CLI receives the encrypted token and
stores it locally. See [CLI Login Flow](apis/functions#cli-login-flow)
for the cryptographic details.

For non-interactive setups, generate a token via the web UI
(Settings → API Keys, prefix `sbp_`) and export it:

```bash
export SUPABASE_ACCESS_TOKEN=sbp_…
```

## Linking a Project

Trex exposes a single synthetic project (`ref = trexsqldefaultlocall`):

```bash
trex link --project-ref trexsqldefaultlocall \
          --use-api http://localhost:8001
```

This populates `supabase/config.toml` with the Trex endpoints and pulls down the
current auth / database / storage configuration.

## Common Commands

### Edge Functions

```bash
# Create a function locally
trex functions new hello-world

# Deploy a function
trex functions deploy hello-world

# Invoke a deployed function
trex functions invoke hello-world --data '{"name":"world"}'

# Download a deployed function (returns the ESZIP bundle, or source if no bundle)
trex functions download hello-world

# Delete a function
trex functions delete hello-world
```

### Secrets

```bash
trex secrets list
trex secrets set MY_API_KEY=… ANOTHER_VAR=…
trex secrets unset MY_API_KEY
```

Secrets land in `trex.secret` (encrypted) and are injected into every edge
function worker.

### Database Connection

```bash
# Start a one-hour scoped Postgres role and print psql connection info
trex db connect --use-api http://localhost:8001

# Generate Supabase-style TypeScript types from your Postgres schemas
trex gen types typescript --schema public,trex
```

### Configuration

```bash
# Pull the current server config into supabase/config.toml
trex config pull

# Push local changes back
trex config push
```

`config push` updates rows in `trex.setting` for the auth / Postgres / PostgREST
/ storage sections.

## Compatibility Notes

- The `trex start` / `trex stop` commands manage local Docker containers, which
  is unrelated to a remote Trex server — they emulate the local-stack workflow
  inherited from the Supabase CLI fork.
- Several Supabase-specific endpoints are stubbed
  (`/v1/projects/:ref/network-restrictions`, `/billing/addons`, etc.) so commands
  like `trex config push` succeed without the underlying feature being
  implemented.
- Branching, point-in-time recovery, and GoTrue advanced flows are not
  implemented.
- The on-disk config directory is named `supabase/` (not `trex/`) so the upstream
  Supabase CLI can also be used against the same project without re-running
  `init`.
- The Go module path of the source remains `github.com/supabase/cli` (the
  upstream fork path). This affects `go install` invocations but is invisible
  to end users of the published `trex` binary.

## Plugin Source

`plugins/cli/` ships the source of the Trex CLI fork. Build artifacts are not
included in the runtime image — the CLI is a developer tool, not a runtime
service. See [`plugins/cli/README.md`](https://github.com/p-hoffmann/trexsql/tree/main/plugins/cli/README.md)
for build details.
